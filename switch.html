01 · Overview

開放式網路（Open Networking）是什麼

傳統交換機廠商（Cisco、Juniper）採取軟硬體綁死的模式：買了硬體，就只能用原廠附贈的專有作業系統。 鈺登（Edgecore）所屬的智邦集團，走的是軟硬體解耦（Disaggregation）路線——硬體與作業系統各自獨立採購。

◆ Bare-Metal Switch

出廠時交換器內部幾乎是空的，沒有完整作業系統，客戶依照自己的需求選擇要安裝的網路作業系統（NOS）。

◆ ONIE

Open Network Install Environment，安裝在 Bootloader 階段，功能類似電腦的 BIOS/Grub，負責把 NOS 安裝進交換器。

常見的 NOS（Network Operating System）

名稱說明
SONiCSoftware for Open Networking in the Cloud，微軟主導的開源交換機系統，基於 Debian Linux，目前是 AI 數據中心主流，鈺登是 SONiC 的主要硬體支持者。
EcNOSAccton / Edgecore 自研的網路作業系統。
白話理解：ONIE 就像手機的 Recovery Mode，交換機開機時先進 ONIE，再由 ONIE 去網路上抓取並安裝指定版本的 SONiC 或 EcNOS。
02 · Architecture

交換機硬體架構

拆開一台鈺登交換機，主要分成三大區塊：控制面（CPU）、轉發面（ASIC）、以及光模組插槽。

控制面
CPU（x86/ARM）跑 Linux、處理 OSPF/BGP、經 I2C/PCIe 控制下層
轉發面
Switch ASIC（Broadcom Trident/Tomahawk、Marvell）硬體級封包轉發
介面端
ASIC 高速訊號拉到面板插槽（Cage），插上光收發模組
03 · 光學基礎

光纖與連接器基礎

光模組再怎麼厲害，最終還是要靠實體光纖傳光。實習現場最常混淆的就是「單模／多模」和「拋光面」，插錯一個顏色的跳線，訊號直接不通或損耗爆表。

類型纖芯常見波長傳輸距離應用
MMF 多模光纖50/62.5μm（較粗）850nm短距（<100m~500m）機櫃內、資料中心短連接，通常搭配 VCSEL 雷射，成本較低
SMF 單模光纖9μm（較細）1310nm / 1550nm長距（數公里~數十公里）機房間、跨樓層長距離骨幹連接

◆ 連接器：LC vs MPO/MTP

LC：最常見的雙芯接頭，一般 SFP/QSFP 單通道模組用。

MPO/MTP：一次集成 8/12/24 芯，用在 QSFP-DD/OSFP 這類多通道高速模組的 breakout 應用。

◆ 拋光面：UPC vs APC

UPC（藍色接頭）：平面拋光，反射損耗較低但仍有些微反射。

APC（綠色接頭）：8° 角度拋光，幾乎消除反射，常用於類比或高精度長距傳輸。

現場地雷：UPC 與 APC 接頭絕對不能混插——角度不同會造成物理錯位，輕則訊號劣化，重則刮傷光纖端面。第一次領料時務必核對顏色（藍 = UPC，綠 = APC）。
04 · 光學基礎

光學參數與雷射安全

光模組的規格書上永遠會出現這幾個參數，看不懂就沒辦法判斷一條鏈路「通不通、好不好」：

Insertion Loss
插入損耗：訊號通過連接器/接頭後的功率衰減，數值越小越好（一般 < 0.3dB）
Return Loss
回波損耗：訊號被反射回發射端的比例，數值越大代表反射越少、越好
Attenuation
光纖本身每公里的衰減量，SMF 約 0.2~0.35 dB/km
Link Budget
發射功率 − 各項損耗總和，決定這條鏈路最遠能撐多遠

雷射安全等級

光模組內建的雷射看不見（多為紅外光），但功率足以傷害視網膜，實習現場的基本安全守則：

  • 絕對不要用肉眼直視光纖端面或模組發射孔，尤其是「看起來沒有插東西」的活動端口
  • 大多數資料中心用模組屬於 Class 1M，肉眼直視安全，但用放大鏡/ 顯微鏡觀察時仍具危險性
  • 不使用中的光纖端口，務必蓋上防塵蓋（同時防止雷射外洩與灰塵進入）
  • 清潔光纖端面要用專用的光纖清潔筆/清潔布，不可用一般衛生紙或酒精棉片以外的東西擦拭
05 · Transceiver

光模組（Transceiver）基礎

光模組是交換機面對外界的窗口，封裝規格會隨傳輸速率演進：

封裝規格常見速率備註
QSFP28100G四通道 25G，早期高速交換主力
QSFP-DD400GDouble Density，針腳更多、頻寬更高
OSFP400G / 800G散熱空間更大，AI 數據中心主流
MSA（Multi-Source Agreement）：業界共同制定的光模組暫存器地圖，例如 SFF-8472 SFF-8636。工程師必須知道哪個 Address 存溫度、哪個存接收光功率（Rx Power）。
06 · Diagnostics

I2C 匯流排、MSA 暫存器與 DDM/DOM

交換機的溫度、風扇、電源，以及光模組內部的 EEPROM，都是透過 I2C（Inter-Integrated Circuit）匯流排讀取。 光模組製造商會把波長、廠商名稱、序號等資訊寫進 EEPROM，通常掛在 0x50 / 0x51 這兩個硬體地址。

i2cdump -y 1 0x50 — 光模組 EEPROM 前 128 bytes（示意）
廠商資訊 (Vendor Name) 波長 / 序號區段 保留 / 未使用

DDM / DOM 即時診斷讀值

當光路不通或 BER（誤碼率）過高，第一步就是進系統看 DDM 診斷數據：

Temperature
42.3°C
正常 0~70°C
Voltage
3.29 V
正常 3.1~3.5V
Tx Power
-2.1 dBm
正常 -6~2 dBm
Rx Power
-14.8 dBm
⚠ 偏低，需檢查光路
實務判斷：Rx Power 明顯低於正常範圍，通常代表光纖端面髒污、彎折過度，或對端發射功率不足，應優先排查物理連結。
07 · Management Interface

CMIS — 新一代光模組管理介面

SFF-8636 是 QSFP28 世代的暫存器地圖，但到了 QSFP-DD / OSFP 這種 400G/800G 模組，業界改用範圍更大、更有彈性的 CMIS（Common Management Interface Specification）。實習如果碰的是新一代模組，這是比 SFF-8636 更該優先熟悉的規範。

◆ 為什麼需要 CMIS

SFF-8636 的暫存器空間有限，無法應付多通道（8 個 lane）、多種應用模式（AppSel）與韌體更新等新需求，CMIS 定義了分頁（Page）機制擴充暫存器空間。

◆ 分頁（Page）架構

CMIS 用 Page 00h/01h/02h/10h/11h... 把暫存器切成多個頁面，例如 Page 10h 存 lane 控制狀態，Page 11h 存即時 DOM 監控值。

模組狀態機（State Machine）說明
ModuleLowPwr剛插入，低功耗待機狀態
ModulePwrUp開始上電初始化
ModuleReady初始化完成，可設定應用模式（Application）並開始傳輸資料
ModuleFault初始化失敗，需檢查 Page 00h 的 fault 相關暫存器
實習場景：模組插上去 ip link 一直起不來，先別急著換模組——先用 CMIS 工具讀 Module State，確認是卡在 ModuleLowPwr（軟體沒下電）還是 ModuleFault（硬體真的壞了），能省下大量排查時間。
08 · Linux 指令

ip link — 網路介面管理

用來查看與控制交換器上所有實體、虛擬網路卡的狀態（Up 或 Down）。

bash
$ ip link show                    # 列出所有網路卡介面、MAC 地址與 MTU
$ ip link set dev eth0 up          # 啟用名為 eth0 的網卡
實習場景：測試光模組插上去後，對應的交換器連接埠（例如 et1 或 xe1）有沒有順利 Link Up。 如果插上去還是 Down，用這個指令確認該介面是否被軟體 up 起來。
09 · Linux 指令

dmesg — 核心訊息日誌

顯示 Linux 核心（Kernel）的環形緩衝區訊息。所有硬體插拔、驅動程式錯誤、系統異常，都會第一時間寫在這裡。

bash
$ dmesg -T | grep -i optical      # 檢查光模組插拔時，底層有沒有抓到驅動與硬體
$ dmesg -w                         # 即時監控核心訊息
[  842.1123] optical: qsfp1 EEPROM read timeout, retrying...
[  842.1129] optical: qsfp1 temp sensor over threshold

當把 400G 光模組插進插槽，終端機通常會跳出核心 log；若模組有硬體故障（I2C 讀寫失敗、過熱保護），dmesg 會直接跳紅字警告。

10 · Linux 指令

i2cdetect / i2cdump — I2C 匯流排偵測與讀取

交換器內部控制風扇、溫度感測器、電源，以及讀取光模組內部 EEPROM 暫存器，都是透過 I2C 匯流排。

bash
$ i2cdetect -l                    # 列出所有 I2C 巴士（buses）
$ i2cdetect -y 1                  # 掃描 1 號巴士，看上面掛了哪些晶片/光模組
$ i2cdump -y 1 0x50               # 把 0x50 位址的暫存器數值整張印出來
實習場景：如果交換機認不出某個光模組，直接用 i2cdump 撈 0x50 地址的十六進位代碼，對照 MSA 規格書確認寫錯的欄位。
11 · ASIC / SDK

Broadcom SDK 診斷指令

當問題出在轉發面（ASIC）而不是 Linux 層，就要進 ASIC 廠商提供的 SDK 診斷殼層（diag shell），直接跟晶片對話。

bcmcmd
BCM.0> ps                        # 顯示所有連接埠即時狀態（linkscan）
BCM.0> show counters              # 顯示各埠封包計數（含 CRC error、drop）
BCM.0> l2 show                     # 列出 ASIC 硬體 MAC table
BCM.0> port xe1 status             # 查詢單一連接埠的實體層狀態
常見指令用途
psPort Status，最常用的第一道排查指令，看哪個埠 Up/Down
show counters檢查是否有 CRC error、drop、collision 等硬體層級錯誤累積
l2 show / l3 show查詢 ASIC 硬體 MAC table / 路由表，跟軟體層看到的是否一致
phy diag查詢 SerDes/PHY 訊號品質（如 eye height）
觀念釐清：ip link 是 Linux 核心看到的介面狀態；ps/port status 是 ASIC 硬體實際看到的狀態。 兩者不一致時（軟體顯示 Up 但 ASIC 顯示 Down），問題通常出在驅動程式或 SDK 初始化，而不是光模組本身。
12 · Console

Console 連線：跟交換器溝通的第一道門

當交換器還沒設定 IP、或系統開機失敗（Kernel Panic）時，無法透過網路（SSH）連線，唯一的救星是實體 Console 線（RJ45 轉 DB9/USB）序列埠連線。

◆ minicom（Linux）

  1. 插上 USB 轉 Serial 線，ls /dev/ttyUSB* 確認裝置名稱
  2. 輸入 sudo minicom -s 進入設定選單
  3. 選擇 Serial Port Setup，設為 /dev/ttyUSB0
  4. Bps/Par/Bits 設為 115200 8N1
  5. Hardware Flow Control 務必設為 No，否則鍵盤打不出字

◆ PuTTY（Windows）

  1. 裝置管理員確認 USB 轉接線佔用的 COM Port（如 COM3）
  2. 連線類型選擇 Serial
  3. Serial line 輸入 COM3，Speed 輸入 115200
  4. 點選 Open，按 Enter 即可看到登入畫面或 ONIE Bootloader 畫面

13 · 網路概念

VLAN — 虛擬區域網路

在實體交換器上透過軟體切割出多個「邏輯上獨立的廣播域」。不同 VLAN 的設備，在沒有路由器（Layer 3）協助下絕對無法互通。

Port 類型說明
Access Port通常接一般電腦，只屬於單一 VLAN
Trunk Port交換器互連通道，同時傳輸多個 VLAN，並用 802.1Q Tag 區分
測試實務：設定特定 VLAN Tag，驗證交換機 ASIC 有沒有確實做好硬體過濾（Tag Filter），不該通的 VLAN 有沒有確實擋掉。
14 · 網路概念

LACP — 網路聚合協定

將多條實體網路線捆綁在一起，邏輯上合併成一條高頻寬、具備備援能力的虛擬大水管（Port Channel / Aggregation Group）。

◆ 頻寬加倍

兩條 10G 的線綁在一起，邏輯上可提供 20G 頻寬。

◆ 容錯備援 Failover

其中一條實體線被拔掉，流量會在小於 50 毫秒內無縫轉移到另一條線，網路不會中斷。

測試實務：測試網通設備時會瘋狂拔線——一邊跑流量測試儀，一邊手動拔掉 LACP 群組中的某條光纖，觀察封包遺失率（Packet Loss）是否維持在極低範圍。
15 · 網路概念

MTU / Jumbo Frame

乙太網路卡在單一影格（Frame）中能傳輸的最大資料載荷（Payload），預設一般為 1500 Bytes。

在 100G/400G 高速、大資料量的 AI 數據中心裡，封包數量極其龐大。若維持 1500 限制，CPU 光是拆解、處理封包標頭就會過載，因此會把 MTU 放大到 9000 甚至更高。

注意：兩端設備（包含中間的交換器）MTU 設定必須一致。若端點送出 9000 Bytes 且設定不可切割（DF flag），中途交換器 MTU 只設 1500 時，該封包會直接被丟棄（Drop）。
16 · 測試與除錯

光功率計 / OTDR 基本判讀

軟體指令查不出來的問題，最後都要靠實體儀器量測光路本身健不健康。

◆ 光功率計 (Optical Power Meter)

直接量測某一點的光功率（單位 dBm）。最常見用法：拔下模組端的跳線，接上功率計，量測對端實際送過來的光功率，跟模組規格書上的接收靈敏度（Rx Sensitivity）比對。

量測值 > 靈敏度
代表光路正常，問題可能在模組本身或設定
量測值 < 靈敏度
代表光路本身有問題（衰減過大、斷點、端面髒污）

◆ OTDR (光時域反射儀)

對長距離光纖特別有用：從一端發出光脈衝，依反射光回來的時間與強度，畫出整條光纖的「損耗地圖」。可以直接定位斷點在幾公尺處，或哪一段接頭損耗異常大。

實習場景：模組亮紅燈、Link Down，先用光功率計量兩端實際收到的光功率，比對規格書門檻值，往往幾分鐘就能判斷「是模組壞、還是光纖本身斷了」，比盲目換模組有效率得多。
17 · 測試與除錯

Ixia / Spirent 流量測試

光路通了、Link Up 了，不代表這條鏈路在滿載流量下也沒問題。流量產生器（Traffic Generator）就是用來灌爆整條鏈路、驗證交換機在高壓下的表現。

典型測試流程

  1. 1. 拓樸連接：把 Ixia/Spirent 埠透過光纖接到待測交換機的兩個（或多個）連接埠
  2. 2. 設定封包參數：指定封包大小（例如 64/512/1518/9000 bytes）、流量速率（%線速）、封包型態
  3. 3. 開始灌流量：從 A 埠打流量到 B 埠，同時在儀器上即時監看發送/接收計數
  4. 4. 記錄結果：Packet Loss（封包遺失率）、Latency（延遲）、Jitter（延遲抖動）
RFC 2544：業界標準的網路設備效能測試方法，主要驗證四項指標—— ThroughputLatencyFrame LossBack-to-back， 面試或報告中常直接被問「有沒有做過 RFC2544 測試」。
LACP 拔線測試（延伸 §14）：灌著滿載流量的同時，手動拔掉 LACP 群組中的一條光纖，觀察 Packet Loss 是否只出現極短暫的尖峰（理想 < 50ms 內恢復），這是驗證容錯機制最直接的方式。
